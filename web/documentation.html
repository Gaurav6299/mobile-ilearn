1–Overview

MiLearn is a streamlined, mobile version of Blackboard iLearn. It is currently available for Android devices.

1.1–Requirements

MiLearn was proposed to be a mobile version of student course information following the structure implemented for two types of users: instructors and students.

  • Alerts
    • Instructors can make alerts (announcements).
    • Students are notified of alerts and can view their content.
  • Classes
    • Instructors can view the classes they teach.
    • Students can view the classes they are enrolled in.
  • Assignments
    • Instructors can view and creates assignments.
    • Students can review/submit their assignments.
  • Grades
    • Instructors can update grades in the GUI.
    • Students can view their grades.
  • Settings
    • Instructors and students can personalize the visual interface of MiLearn.
    • Both types of users can also customize when alert notifications.

1.2–Development Process

During development of MiLearn, we used a Scrum process of development. Our group would meet up at least once a week on Mondays for discussing our contributions towards the project, as well as planning on what part of the project to do next.

What prevented our Scrum process from degenerating into the bug-fix model was rigor. Each week we would test reliability and correctness by creating various regression tests.

By using the Scrum method we had high visibility of our problems and goals each week.

In addition, we took a modular approach towards development. Each group member focused on a different component of the app. This boosted overall group efficiency as we were able to concentrate on one item to do per week instead of several.

One problem was actually getting each component integrated properly each week. Some file writers and other libraries were not supported by Android and we had to find workarounds.

Previous Java classes we wrote had to be reimplemented in order to function properly with Tomcat.

Scalability of the system was tested after a small sample by increasing the database to 103 users, 20 lectures, 5 labs, 20 instructors, 5 lab instructors, and 50 assignments.

2–User Manual

This is a quick reference for using the MiLearn app.

For those who wish to know how the MiLearn app, please look here first.

2.1–Logging In/Logging Out

When MiLearn is first opened, the Login screen is brought up. Here, the user will need to insert their username (i.e. UCR NetID) and password.

2.2–Home Screen

Once the user has logged in, they are brought to the Home Screen. This will initially display the "Welcome to MiLearn" screen.

The Home Screen should then display the user's name and any alerts that they may have.

Alerts are coded such that high priority alerts are in red, medium priority in yellow, and low priority in green.

The timing of each priority can be set in the Settings Screen.

2.3–Classes Screen

The Classes Screen displays an accordion for all of the user's classes for the current quarter. The header of each collapsible contains the course number followed by the course section. Inside of each collapsible is the full name of the course.

If the user is a student, then they will see what class(es) that they are enrolled in. In addition the name(s) of the instructor(s) are displayed inside the collapsible.

If the user is an instructor, then they will see what class(es) they are teaching. The collapsible content also includes a list of students that are enrolled in that course.

2.4–Assignments Screen

The Assignments Screen displays an accordion with the assignments ordered by due date (from sooner to later). Each collapsible header has the course that the assignment is for followed by the assignment title.

Students can view what assignments are due for all classes that they are enrolled in. They can also upload a file to turn in for an assignment. Assignment uploads can be done multiple times, but the instructor will only see the most recent submission. Currently, students can only upload one file at a time.

Instructors can create new assignments or edit assignments for any class they teach. This is respectively done through an already existing assignment collapsible or through the "New Assignment" collapsible at the bottom of the list.

2.5–Grades Screen

The Grades Screen displays an accordion with the grades for each assignment in each class.

For students, each collapsible is for a course that they are enrolled in. The collapsible heading indicates which course the student they will be looking in while the content displays (in chronological order), the grades for each assignment.

Instructors each collapsible is for an assignment that they have assigned and is ordered chronologically. Inside each collapsible is the name of the assignment, how many points it was worth, and a table of students' grades for the assignment. The total points and each student grade can be edited.

2.6–Settings Screen

The Settings Screen allows all users to configure the theme of their user interface and the time for priority levels.

A quick guide for themes:

Theme Description
Default Generic gray/chrome theme.
Dark Generic dark theme.
Highlander Show UCR school spirit.
Go Blind! For trolls and schmucks only.
Super! Basic! ... why?

Priority levels should reflect the user's feelings about how soon they should begin an assignment or how late they should be when finishing one.

3–Design/Development Process

During the development of MiLearn, each team member took responsibility for developing a component of the MiLearn app. Each member had their own rationale and design process regarding their own component.

The MiLearn app project was split between members such that:

Candice Bantejac
Worked on the user, classes, database, Tomcat, and Java servlets of Milearn.
Vincent Chang
Worked on the Android portion of MiLearn.
Rebecca Hom
Worked on the user interface of MiLearn.
Krissi Mena
Worked on S3, databases associated with assignments, grades, and submissions, and file upload aspects of MiLearn.

The following sections are broken up further into what each member of the project rationale about the portion they picked to work on, what worked, and what features they designed and implemented.

3.1–Candice Bentejac

Rationale

Candice decided to work on the users, classes, login/logout module and the associated PostgreSQL database (hosted on the cloud using Amazon Web Services RDS). Later on during the project, the team had to handle data from several HTML forms to Java: Java Servlets have thus been used, as well as Amazon Web Services EC2, which allowed us to create a virtual machine on which we could install our server. Amazon Web Services S3 has also been used to store data that we could easily access from the cloud.

The goal of the project being to create an Android app, the backend of the application has been entirely coded using Java. We wanted MiLearn to be a complete and easy-to-use alternative to the BlackBoard iLearn mobile app. The users would thus have access to their Classes, Assignments, Grades and would be able to set up alerts with three different priority levels.

Since Rebecca used Javascript to develop the frontend, we needed a way to give her access to the data that we were parsing in Java. To do so, we decided to use JSON files that would be created with Java, sent to the cloud (more precisely, to an S3 bucket) and then parsed by Rebecca's JavaScript.

Candice's tasks consisted of creating Java objects that would represent the users and the classes. Users were divided into three categories: the Students, the Instructors, and the Teaching Assistants. Something similar was done with the Classes, which were divided into Lectures, Laboratories, and Discussions. Users can, depending on their status, be enrolled in a class or teach a class. The Teaching Assistant are both enrolled in and teaching a class.

Another of Candice's goals was to implement the login/logout module.The login module required a direct interaction between the user and the app. For this reason, it was developed in collaboration with Vincent, since he in charge of integrating the code in an Android environment. Vincent designed and implemented the login page, and Candice handled the login verification.

Candice also set up an Amazon Web Services RDS database that used PostgreSQL. She also collaborated with Krissi to maintain this database, and Krissi populated it with some user data.

Despite the resources already available, we very quickly needed to get data from the users to the backend, and decided to use HTML forms to do so. The users would only have to fill in those forms and submit them. Using Java Servlets, we can parse those HTML forms in Java and respond to the user's request - safely storing secure data while avoid corrupting the database. Candice consequently created an Amazon Web Services EC2 instance, setting up a virtual machine and installing Tomcat on it. Servlets were then implemented, stored, configured and mapped on this EC2 instance, on which Tomcat runs continuously.

What Worked

Candice implemented User and Classes classes, which were linked to the Login class, that Vincent integrated into the Android environment. Those Java classes allowed JSON files to be automatically generated and stored as soon as a user was logged in. Many libraries were integrated into the project to handle the database requests and to use Java Servlets. Java Servlets were used to treat data sent by users, and to regenerate certain JSON files when the context justified it. Amazon Web Services tools that were constantly used were also integrated to the code.

Retrieving data from the database and generating the corresponding JSON files that could be read by Rebecca's code was the main challenge, since all the the user interface relied on the correctness of those JSON files.

Features Designed and Implemented

  • Users and Classes
    • Creation and setting up of a RDS PostgreSQL database
      • Candice created a class called DBManager that contained basic and more advanced SQL operations.
      • Krissi populated the database.
      • Made User and Class classes interact with the database.
    • Creation and setting up of a S3 instance
      • Rebecca created the first bucket and Krissi created a class making the use of this instance (especially the upload/downloads operations) way easier and clearer.
      • The Amazon Web Services libraries were used a lot to send freshly generated JSON files to the cloud.
    • Creation and setting up of a EC2 instance
      • Installed Tomcat 8 on an Amazon virtual machine
      • Created Java Servlets that were parsing data retrieved from HTML forms and treating it afterwards.
      • Created a Java Servlet that handles uploading files to S3.
      • Handled JSON files regeneration when the database is updated.
  • Login module
    • Created login class that uses User and Class classes.
      • "Merged" the JSON files generation for User and Class.
      • Automated JSON generation when a user logs in.
      • Checked the user's input and determined whether they had the correct credentials or not. All associated data was generated if the credentials were accepted. Otherwise the connection was rejected.
    • Vincent integrated this module to the Android environment.

3.3–Rebecca Hom

Rationale

Rebecca decided to work on the user interface due to having previous experience working with implementing designs using JavaScript. Although it was possible to use Android development to create the user interface, using jQuery Mobile was allowed for a more efficient use of her skillset and it came with icons that could be used on the app.

Since MiLearn is a simplified version of Blackboard iLearn, Rebecca arranged the essential functions into an intuitive design. This accomplished two things: retaining the simplicity of the app and splitting up each function into its own section. In turn, each of these sections could be styled and displayed as distinctly as possible. For the MiLearn app, the user would be able to access a Home screen, Classes, Assignments, Grades, and Settings. Each of those sections have JavaScript functions that generate the data to display from the database.

It was also decided that the data displayed for a user would be taken from fixed JSON files. Each of these JSON files had fixed names and attributes for efficient parsing and displaying using JavaScript. These files would also be consistent for each user and user level which allowed for easily cached information and reduced the need to do new queries to the database.

Also it was decided that a set of collapsibles would be used for Classes, Assignments, and Grades. This would hide information pertaining to a topic heading. In turn, this would be more efficient for the user as they would be able to scroll through headings to get what they needed instead of (potentially) large blocks of information.

Alerts were added to the Home screen as a way to increase MiLearn's functionality as a mobile organizer. The user would be able to set three different alarm priority levels. These could be used as a buffer for the user to guage how much time they have left to do an assignment. In addition, themes for the user interface were included. This would allow the user to somewhat customize MiLearn for their own use.

Users were also split into two groups: students and instructors. The HTML page was then split into separate dividers to allow for either of those groups or a combination of them to use the app. HTML forms were used as a security feature for users modifying the database. This was to allow for input sanitation to prevent users from corrupting the database.

What Worked

Rebecca implemented an HTML page that contained all of the screens for main MiLearn app that allowed for the screens to change using JavaScript.

In addition changing the screens, JavaScript was also used to parse the JSON files to display the data elements into their certain arrangements depending on what screen was used.

JavaScript was also used to interact with Java to retrieve user information and move in and out from the Android app.

Features Designed and Implemented

  • MiLearn page
    • Home Screen
      • Welcome screen with image.
      • Alerts for assignments that are due within a certain period of time.
    • Classes
      • Created a screen that displayed the current quarter of the class.
      • Showed the classes that the user was enrolled in and/or teaching in a collapsible set.
    • Assignments
      • Created a file upload form for student users to turn in files from their mobile device.
      • Created an HTML form for instructor users to either create new assignments or modify existing assignments.
    • Grades
      • Created a tabular HTML form so that instructor users could input grades for each assignment they created.
      • Allowed student users to view the overall number of points they have for a class and the grade for individual assignments.
    • Settings
      • Used JavaScript to allow users to change their user interface theme and the different alarm priority levels.
  • Crumb Lords website
    • Added in HTML version of documentation.
    • Built site to present information about MiLearn project.
